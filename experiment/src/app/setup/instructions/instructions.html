Frontend Component Libraries - Entwicklungsexperiment

📅 Geschätzte Dauer: 45-60 Minuten 📈 Schwierigkeit: Mittel 👥 Zielgruppe: Frontend-/ Fullstack-Entwickler

🎯 Experiment-Übersicht

Dieses Experiment vergleicht zwei Ansätze für Frontend Component Libraries im Hinblick auf Entwicklungseffizienz bei kundenspezifischen Anpassungen:

🎨 Angular Material

Traditionelle Component Library mit Theming-System

  • Etablierter Standard
  • Umfassende Dokumentation
  • Material Design Guidelines

⚡ spade

Copy-Paste Library mit vollständiger Code-Kontrolle

  • "Own your code" Prinzip
  • Unbegrenzte Anpassbarkeit
  • Keine Breaking Changes

📝 Aufgaben-Übersicht

Task 1: Button Component

⏱️ ~15 Minuten 📊 Einfach

Szenario: Corporate Design-Anpassung für Automotive-Kunden

  • Outline-Variante: Transparenter Hintergrund, farbiger Border
  • Farbschema: Primary #1E3A8A, Secondary #F59E0B
  • Icon-Only Button: Plus-Icon Support mit ARIA Labels
  • Alle States: Default, Hover, Focus, Disabled

Task 2: Input Field

⏱️ ~20 Minuten 📊 Mittel

Szenario: Formular-Komponente für öffentliche Verwaltung

  • Zeichenzähler: Live-Counter rechts im Input (max 250)
  • Visuelles Feedback: Grün <200, Orange 200-240, Rot >240
  • Integration: Mit bestehenden Validation States
  • Real-time Updates: Counter aktualisiert beim Tippen

Task 3: Dropdown

⏱️ ~30 Minuten 📊 Komplex

Szenario: Multi-Select für E-Commerce Dashboard

  • Such-Funktionalität: Live-Filter für 50+ Optionen
  • Multi-Select Chips: Rounded Chips mit Remove-Button
  • Keyboard Navigation: ESC löscht Suche
  • Performance: Smooth Interaction bei 50+ Items

⚠️ Wichtige Hinweise

🕐 Zeitmessung

Notiere Start- und Endzeiten manuell für jede Implementierung. Stoppe nach dem Zeitlimit, auch wenn nicht fertig.

⏰ 30-Minuten-Regel

Nach 30 Minuten Gesamtzeit pro Aufgabe kannst du das Experiment beenden. Teile dann deine subjektiven Eindrücke zur Komplexität im Feedback mit.

🔍 Beide Implementierungen

Arbeite an Material UND spade für jeden gewählten Task. Vergleiche die Ansätze direkt.

📊 Datensammlung

Dokumentiere Herausforderungen, Zeitaufwand und subjektive Eindrücke für die Auswertung.

📁 Datei-Bearbeitungsrichtlinien

🎨 Angular Material

Wo bearbeiten:

  • task-[component]/task-[component].scss - Styling-Anpassungen
  • task-[component]/task-[component].ts - Custom Logik Implementierungen
  • styles.scss - Theme-Anpassungen

Approach: CSS-Overrides, Angular Material Theming und Logik in der task-[component]/task-[component].ts

⚡ spade Components

Wo bearbeiten:

  • components/spade-[component]/ - Vollständige Code-Anpassbarkeit
  • tokens/ - Design Tokens anpassen
  • task-[component]/ - Anwendung neuer Properties, Overrides etc.

Approach: Direkte Bearbeitung des Component-Codes in components/spade-[component]/ oder Anpassungen in task-[component]/, tokens/

🔧 Pre-Setup Information

Die spade-Komponenten wurden bereits über die CLI kopiert (simuliert spade get button). Du kannst sie direkt bearbeiten, als wären es deine eigenen Projekt-Komponenten.

📈 Was wird gemessen?

Quantitative Metriken

  • Implementierungszeit pro Ansatz
  • Zeilen geänderter Code
  • CSS-Komplexität und Overrides
  • Vervollständigungsgrad der Requirements

Qualitative Bewertung

  • Subjektive Schwierigkeit (1-5 Skala)
  • Developer Experience
  • Spezifische Herausforderungen
  • Präferenzen und Begründungen
