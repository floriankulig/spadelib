Frontend Component Libraries - Development Experiment

📅 Estimated Duration: 45-60 minutes 📈 Difficulty: Medium 👥 Target Group: Frontend / Fullstack Developers

🎯 Experiment Overview

This experiment compares two approaches for frontend component libraries in terms of development efficiency for custom adaptations:

🎨 Angular Material

Traditional component library with theming system

  • Established standard
  • Comprehensive documentation
  • Material Design Guidelines

⚡ spade

Copy-paste library with full code control

  • "Own your code" principle
  • Unlimited customizability
  • No breaking changes

📝 Task Overview

Task 1: Button Component

⏱️ ~15 minutes 📊 Easy

Scenario: Corporate design adaptation for automotive client

  • Outline variant: Transparent background, colored border
  • Color scheme: Primary #1E3A8A, Secondary #F59E0B
  • Icon-only button: Plus icon support with ARIA labels
  • All states: Default, Hover, Focus, Disabled

Task 2: Input Field

⏱️ ~20 minutes 📊 Medium

Scenario: Form component for public administration

  • Character counter: Live counter on the right in input (max 200)
  • Visual feedback: Green <160, Orange 160-190, Red >200
  • Integration: With existing validation states
  • Real-time updates: Counter updates while typing

Task 3: Dropdown

⏱️ ~30 minutes 📊 Complex

Scenario: Multi-select for e-commerce dashboard

  • Search functionality: Live filter for 50+ options
  • Multi-select chips: Rounded chips with remove button
  • Keyboard navigation: ESC clears search
  • Performance: Smooth interaction with 50+ items

🔄 Experiment Process

1

Setup (5 min)

  • Make sure ng serve is running
  • Open your preferred IDE
  • Prepare time tracking (stopwatch/timer)
2

Implementation (Max. 30 min)

  • Choose 1-3 tasks (order doesn't matter)
  • Both implementations per task
  • Track time for each implementation
  • 30-min limit: After that, provide subjective complexity assessment
3

Feedback (10 min)

  • Fill out feedback form
  • Document times and experiences
  • Provide subjective assessments

⚠️ Important Notes

🕐 Time Tracking

Record start and end times manually for each implementation. Stop after the time limit, even if not finished.

⏰ 30-Minute Rule

After 30 minutes total time per task, you can end the experiment. Then share your subjective impressions about complexity in the feedback.

🔍 Both Implementations

Work on Material AND spade for each chosen task. Compare the approaches directly.

📊 Data Collection

Document challenges, time spent, and subjective impressions for evaluation.

📁 File Editing Guidelines

🎨 Angular Material

Where to edit:

  • task-[component]/task-[component].scss - Styling adjustments
  • task-[component]/task-[component].ts - Custom logic implementations
  • styles.scss - Theme adjustments

Approach: CSS overrides, Angular Material theming and logic in the task-[component]/task-[component].ts

⚡ spade Components

Where to edit:

  • components/spade-[component]/ - Complete code customizability
  • tokens/ - Adjust design tokens
  • task-[component]/ - Application of new properties, overrides etc.

Approach: Direct editing of component code in components/spade-[component]/ or adjustments in task-[component]/, tokens/

🔧 Pre-Setup Information

The spade components have already been copied via CLI (simulating spade get button). You can edit them directly as if they were your own project components.

📈 What is being measured?

Quantitative Metrics

  • Implementation time per approach
  • Lines of changed code
  • CSS complexity and overrides
  • Completion rate of requirements

Qualitative Assessment

  • Subjective difficulty (1-5 scale)
  • Developer experience
  • Specific challenges
  • Preferences and reasoning
